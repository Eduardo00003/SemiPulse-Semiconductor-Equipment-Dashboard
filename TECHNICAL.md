# SemiPulse — Technical Notes

## Project Overview
SemiPulse is a predictive maintenance dashboard for simulated semiconductor manufacturing equipment.

It combines:
- data ingestion,
- data validation,
- SQLite persistence,
- Pandas feature engineering,
- scikit-learn risk scoring,
- Streamlit dashboarding,
- CSV exports,
- Docker-based local execution.

The project is designed as a portfolio-ready demonstration of applied data engineering and machine learning for equipment maintenance analytics.

Important limitation:
- All datasets are simulated.
- Model results do not represent real semiconductor factory performance.
- Any metric such as 84 percent recall should be described as simulated-data performance only.

---

## Stack
- Language: Python 3.11+
- Dashboard: Streamlit
- Data processing: Pandas, NumPy
- Machine learning: scikit-learn
- Model persistence: joblib
- Database: SQLite
- Charts: Matplotlib and Streamlit-native charts
- Testing: pytest
- Config: environment variables and `.env`
- Containerization: Docker and docker-compose

---

## High-Level Architecture
```text
CSV / generated sample data
        ↓
validation + cleaning
        ↓
SQLite tables
        ↓
feature engineering
        ↓
scikit-learn model training / loading
        ↓
risk predictions + model metrics
        ↓
Streamlit dashboard + exports
```

The app should keep heavy logic outside Streamlit page files.

Recommended separation:
- `app/` contains Streamlit UI.
- `semipulse/` contains reusable pipeline, model, database, and plotting modules.
- `data/` contains sample/raw/processed files.
- `db/` contains SQLite database and schema.
- `models/` contains trained model artifacts and metadata.
- `tests/` contains pytest tests.

## Core Domain Objects

### Machine
A semiconductor equipment asset.

Key fields:
- `machine_id`
- `machine_type`
- `manufacturer`
- `facility_area`
- `install_date`
- `status`
- optional `criticality`

### Sensor Reading
A time-series measurement from a machine.

Key fields:
- `reading_id`
- `machine_id`
- `timestamp`
- `temperature`
- `vibration`
- `pressure`
- `power_draw`

### Maintenance Record
A historical maintenance event.

Key fields:
- `maintenance_id`
- `machine_id`
- `maintenance_date`
- `maintenance_type`
- `downtime_hours`
- optional severity, cost, notes, resolved flag

### Defect Record
A production-quality or defect signal associated with a machine.

Key fields:
- `defect_id`
- `machine_id`
- `timestamp`
- `defect_count`
- `batch_id`
- optional defect type, severity, yield loss percentage

### Machine Feature Row
A machine-level row used for model training and prediction.

Key fields:
- `machine_id`
- sensor aggregate features
- rolling-window features
- maintenance recency features
- downtime features
- defect features
- metadata features
- target label if available

### Risk Prediction
A prediction generated by the model.

Key fields:
- `prediction_id`
- `machine_id`
- `model_run_id`
- `risk_score`
- `predicted_failure_flag`
- `risk_level`
- `prediction_timestamp`

## Data Pipeline Flow

### 1. Load Data
Data can come from:
- generated sample files,
- uploaded CSV files,
- existing SQLite tables.

Expected dataset files:
- `machines.csv`
- `sensor_readings.csv`
- `maintenance_records.csv`
- `defect_records.csv`

### 2. Validate Data
Validation should check:
- required columns,
- timestamp parsing,
- numeric field parsing,
- duplicate IDs,
- orphan machine IDs,
- missing values,
- invalid categories.

Validation should not crash the whole app for one bad row. It should return a structured issue report.

### 3. Clean Data
Cleaning should standardize:
- column names,
- machine IDs,
- timestamps,
- numeric values,
- categorical values,
- missing values.

### 4. Store in SQLite
Cleaned data should be written to SQLite using transaction-safe helpers where practical.

### 5. Generate Features
The feature pipeline should produce `machine_features`.

### 6. Train / Load Model
The ML pipeline should train or load a scikit-learn model.

### 7. Generate Predictions
Predictions should include risk score, predicted failure flag, and risk level.

### 8. Display Dashboard
Streamlit reads from SQLite and model output tables.

## Machine Learning Design

### Model Type
The MVP should use scikit-learn.

Recommended baseline:
```python
RandomForestClassifier(
    n_estimators=200,
    random_state=42,
    class_weight="balanced"
)
```

Possible alternatives:
- Logistic Regression
- Gradient Boosting Classifier

### Target Label
The binary target should represent whether a machine is likely to require maintenance or experience failure within a configured prediction window.

Default:
- 14-day prediction window

Example label:
```text
1 = machine has high-risk maintenance/failure event within next 14 days
0 = no high-risk event within next 14 days
```

### Evaluation Metrics
Required:
- recall
- precision
- F1
- accuracy
- confusion matrix

Optional:
- ROC-AUC when probability outputs and class balance allow it

### Why Recall Matters
For predictive maintenance, missing a truly high-risk machine can be costly. Therefore, the dashboard should explain recall and emphasize it over raw accuracy.

### Model Artifacts
Recommended files:
- `models/risk_model.pkl`
- `models/model_metadata.json`

`model_metadata.json` should include:
- model type,
- model run ID,
- training timestamp,
- feature columns,
- prediction window,
- random seed,
- metrics,
- simulated-data warning.

## Security and Reliability Notes
- Do not store secrets in source control.
- Do not log secrets.
- Use `.env.example` for config documentation.
- Handle missing files gracefully.
- Handle invalid CSVs gracefully.
- Keep sample data small enough for GitHub unless intentionally excluded.
- Add clear warnings that the data is simulated.

## Known Limitations for MVP
- Uses simulated data only.
- SQLite is intended for local/demo use, not high-concurrency production use.
- Model performance depends on generated data patterns.
- The app does not prove real semiconductor equipment failure prediction accuracy.
- Streamlit is sufficient for portfolio/demo usage but is not a full production web architecture.
